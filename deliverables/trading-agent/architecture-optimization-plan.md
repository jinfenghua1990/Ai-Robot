# AIROBOT 9000 端口架构优化方案

> 生成日期: 2026-07-10 | 基于全天全链路巡检数据

---

## 现状总览

| 指标 | 数值 |
|------|------|
| 后端路由数 | 274+ 个 API 端点 |
| 数据采集器 | 21 个独立模块 |
| 数据库表 | 50 张（～944MB） |
| 前端组件 | 394 个（主前端 137 + DSA 257） |
| 运行中子进程 | 3 个（主后端 9000 + DSA 8000 + LiteLLM 8001） |
| 每天数据写入 | 实时资金流 34 万行、Tick 21 万行 |

## 发现的问题

### P0 — 优先级最高

#### 1. 资金流双表并行（已部分修复）
| 表 | 行数 | 大小 | 谁写 | 谁读 |
|---|---|---|---|---|
| `realtime_stock_flow` | 157 万 | **434 MB** | 多源采集+交叉验证 | `/api/realtime/stock-flow-detail` |
| `stock_money_flow_realtime` | 2.1 万 | **5 MB** | emdatah5 + 跨写 | SSE → 前端实时面板 |

同一类数据两种写入方式，后者只是前者的子集。今天已加跨写，但根源上还是两套。

#### 2. 妙想 API 无缓存、频繁超时
`/api/trading/positions`、`/api/trading/balance`、`/api/trading/orders` 都实时请求妙想云 API。妙想不稳定时整个交易模块挂死。**今天因此导致了 3 次后端崩溃。**

#### 3. DSA 独立进程增加死锁风险
DSA 后端（8000）独立运行，主后端通过反向代理转发。尝试在代理层加桥接逻辑时，**多次导致主后端完全挂死**（进程在、不响应），因为 httpx 自调用 + uvicorn 单进程阻塞。

### P1 — 值得优化

#### 4. Hermes 前端已成死代码
Hermes 的**后端 API 已停用、数据库已迁移**，只剩前端静态文件还在服务。占用了 `_hermes/` 路由但无人使用。

#### 5. Vibe 数据不落库
10+ 个 Vibe 端点每次实时请求东财，不做任何缓存。每天重复拉取财报、公告、研报、新闻，浪费带宽也增加东财源压力。**用户曾明确问"数据有没有进数据库"就是这个意思。**

#### 6. 21 个采集器职能重叠
- `emdatah5_collector.py` vs `realtime_collector.py` → 都在做实时资金流
- `tdx_collector.py` vs `extended_collectors.py` → 都在做多源价格采集
- 多个 Tushare 降级互相抢速率配额（已看到 `40203 频率超限` 日志）

### P2 — 锦上添花

#### 7. 三套前端风格不统一
主前端（React SPA）、DSA（React TypeScript SPA）、Hermes（独立 SPA）是三套独立构建产物。样式、组件、路由互不共享。

#### 8. 数据库无定期清理策略
`realtime_stock_flow` 每天 34 万行写入，按此速度每月增加 **1000 万行**、约 3GB。无自动清理策略。

## 优化方案

以下分三种方案，从轻到重：

---

### 方案 A：轻量优化（1-2 天，推荐立刻做）

| # | 优化项 | 操作 | 效果 |
|---|--------|------|------|
| A1 | **合并资金流写入** | 废弃 `emdatah5` 采集路径，全部走 `realtime_collector`（有多源验证），只写 `realtime_stock_flow` | 省 5MB 表，采集代码减 1 个文件 |
| A2 | **妙想 API 本地缓存** | 在 `trading.py` 加 30 秒内存缓存（`cachetools.TTLCache`），妙想超时用缓存兜底 | 消除交易模块挂死风险 |
| A3 | **启用 DB 清理任务** | 定时任务删除 `realtime_stock_flow` 超过 30 天的快照 | 每月省 3GB |
| A4 | **前端资金流统一数据源** | SSE 实时面板改读 `realtime_stock_flow` 而非 `stock_money_flow_realtime` | 统一数据源，前端看到交叉验证后的数据 |

**工作量**: 4 个改动，约 100 行代码

---

### 方案 B：中等优化（3-5 天，推荐做）

在 A 的基础上：

| # | 优化项 | 操作 | 效果 |
|---|--------|------|------|
| B1 | **移除 Hermes 前端** | 删除 `backend/static/hermes/` 目录 + 相关路由 | 省 12MB 静态资源，减少路由混乱 |
| B2 | **Vibe 数据加本地缓存** | 用 `cachetools.TTLCache` 给 Vibe 的 live 端点加 5-30 分钟 TTL 缓存 | 减少 90% 重复请求，东财源压力降低 |
| B3 | **采集器合并** | `emdatah5_collector` 合并进 `realtime_collector`，`extended_collectors` 中的价格采集合进 `astock_collector` | 21 → 16 个文件 |
| B4 | **DSA 桥接稳定化** | portfolio bridge 改用`cachetools`定期拉取（非请求时触发），彻底消除代理层阻塞风险 | 消除死锁风险 |

**工作量**: 5 个改动，约 300 行代码

---

### 方案 C：重度优化（1-2 周，长期工程）

在 A+B 的基础上：

| # | 优化项 | 操作 | 效果 |
|---|--------|------|------|
| C1 | **DSA 合并入主后端** | 把 DSA 的 `server.py` 路由移植到主后端 `main.py` 中，关闭 8000 端口进程 | 省 15MB 内存，消除代理层，消除死锁风险 |
| C2 | **三前端合并为一** | 将 DSA/Hermes 页面作为主前端的新路由，统一构建、统一样式 | 省 30% 前端代码，统一用户体验 |
| C3 | **API Gateway 统一前缀** | 所有 API 统一到 `/api/*` 前缀下，去掉 `dsa_api_proxy` 特殊路由 | 路由从 274 精简到约 180 |
| C4 | **合并 LiteLLM 进主进程** | 将 LLM 网关能力直接集成到主后端 | 省 8MB 内存，省一个子进程 |

**工作量**: 12+ 个改动，约 1000 行代码

---

## 我的推荐

### 立刻做：方案 A（1-2 天）
解决最痛的 4 个问题——资金流双表、妙想超时挂死、数据库膨胀、前端数据源分裂。

### 顺势做：方案 A+B（3-5 天）
顺手清理 Hermes 死代码、Vibe 缓存、采集器合并。**性价比最高的集合**——改动不大，效果明显。

### 以后做：方案 C（1-2 周）
等有整块时间时考虑 DSA 合并和前端统一。目前 DSA 独立进程虽有缺点但运行稳定，不急着动。
